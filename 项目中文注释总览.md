## 1. 项目是干什么的（先建立全局认知）

这是一个 **GitHub Actions 自动构建 Android GKI 内核** 的仓库，目标产物主要有：

- `AnyKernel3.zip`（可直接刷）
- `boot.img / boot-gz.img / boot-lz4.img`（适配不同内核镜像压缩格式）
- 可选附带 **KernelSU 管理器 APK** 与 **SUSFS 模块**

内核特性组合（核心卖点）：

1. KernelSU 多变体（Official / Next / MKSU / SukiSU）
2. SUSFS 隐藏能力补丁
3. BBR / Westwood / 代理优化（TPROXY、ipset、IPv6 NAT）
4. 可选 LZ4KD / LZ4K_OPLUS 的 ZRAM 扩展
5. 可选 Baseband-guard（BBG，LSM 层）

### 2.3 `hmbird_patch.c`

这是一个早期启动阶段执行的内核补丁模块（`early_initcall`）：

- 查找设备树节点 `/soc/oplus,hmbird/version_type`
- 读取属性 `type`
- 当值是 `HMBIRD_OGKI` 时，强制改成 `HMBIRD_GKI`

你现在文件里我已经补了逐段中文注释，重点是：

- 每一步在干什么
- 为什么失败时仍返回 `0`（不中断启动）
- 为什么要 `of_node_put`（引用释放）

## 3. GitHub Actions 总体架构（最重要）

整个工作流是“**调度层 -> 版本矩阵层 -> 核心构建层 -> 发布层**”四层结构：

1. **调度层**：`test_release.yml`（手动选择编译目标）
2. **版本矩阵层**：`kernel-a12-5.10.yml` 等（定义每条内核线要编哪些 sublevel）
3. **核心构建层**：`gki-kernel.yml`（真正拉源码、打补丁、编译、打包）
4. **发布/辅助层**：`get-manager.yml`、`Auto_build.yml`、release 逻辑

## 4. 每个 workflow 的中文注释

## 4.1 `.github/workflows/test_release.yml`（总入口）

**定位：手动触发的一站式总控流程。**

### 输入参数含义

- `kernels_to_build`: 要编译哪些内核线（逗号分隔或 `all`）
- `kernelsu_variant`: KSU 变体
- `kernelsu_branch`: Stable / Dev
- `use_zram` / `use_kpm` / `use_O2` / `BBG`: 功能开关
- `get_manager`: 是否同时获取最新管理器

### job 流程

1. `get_manager`：调用 `get-manager.yml` 取 KSU 版本号、SUSFS 版本号、管理器/APK
2. 并行触发各版本线 job：
   - `build-kernel-a12-5-10`
   - `build-kernel-a13-5-10`
   - `build-kernel-a13-5-15`
   - `build-kernel-a14-5-15`
   - `build-kernel-a14-6-1`
   - `build-kernel-a15-6-6`
3. `check-build-status`：检查“应跑的 job 是否成功”
4. `trigger-release`：汇总 artifacts -> 生成 tag -> 创建 release -> 上传资产

> 核心理解：`test_release.yml` 本身不编译内核，它是编排者。

---

## 4.2 `.github/workflows/kernel-a*.yml`（版本矩阵层）

对应文件：

- `kernel-a12-5.10.yml`
- `kernel-a13-5.10.yml`
- `kernel-a13-5.15.yml`
- `kernel-a14-5.15.yml`
- `kernel-a14-6.1.yml`
- `kernel-a15-6.6.yml`

**定位：只做两件事**

1. 用 `matrix.include` 定义要编译的 `sub_level + os_patch_level`
2. `uses: ./.github/workflows/gki-kernel.yml` 把参数传给核心编译器

也就是说，这些文件是“版本清单配置文件”。

---

## 4.3 `.github/workflows/gki-kernel.yml`（核心编译引擎）

这是仓库里最关键、最长的脚本。可按 10 个阶段理解：

1. **环境准备**：释放 runner 空间、安装 ccache、缓存工具链
2. **工具准备**：拉取 `build-tools`、`mkbootimg`、设置 avbtool 路径
3. **源码准备**：repo init/sync AOSP kernel common，对 deprecated 分支做兼容处理
4. **功能源码注入**：加入 KernelSU / SUSFS / 可选 BBG
5. **补丁阶段**：SUSFS patch、hooks patch、zram patch、隐藏特征 patch、分支兼容修复
6. **defconfig 写入**：KSU/SUSFS/网络/代理/zram/bbr/westwood/O2 等配置写入
7. **构建执行**：build.sh 或 bazel 两套路径，带 retry
8. **镜像处理**：可选 KPM patch Image，生成 Image / Image.lz4 / Image.gz
9. **打包产物**：生成不同压缩格式 AnyKernel3 zip + boot.img 系列
10. **上传 artifacts**：供后续 release job 下载与汇总

### 你最常改的几个位置

- 改要支持的分支/patch：`克隆 AnyKernel3 和其他依赖项` 与补丁阶段
- 改编译优化：`编译优化` 相关 step
- 改 defconfig 默认能力：`添加SUSFS配置设置`、`添加代理性能优化` 等 step
- 改产物命名：`创建不同格式的ZIP文件` 与 boot img copy 命名

---

## 4.4 `.github/workflows/get-manager.yml`（版本与管理器获取器）

核心作用：

1. 根据 KSU 变体拉对应仓库
2. 计算 `KSU_VERSION`
3. 拉取 SUSFS module.prop 拿 `SUS_VERSION`
4. 下载对应 manager 构建产物（APK）
5. 下载 SUSFS 模块产物
6. 把版本号作为 workflow 输出给 release 使用

> 你 release 标题里的版本号，主要就来自这里。

---

## 4.5 `.github/workflows/Auto_build.yml`（自动监控触发）

逻辑：

1. 每 7 天检测 SukiSU-Ultra 主分支最新 commit SHA
2. 对比仓库 `sha` 分支保存的旧 SHA
3. 有更新则写回 `latest_sha.txt`
4. 自动 dispatch `test_release.yml` 开始编译

> 这是“无人值守定时跟进上游”的自动化入口。

---

## 4.6 `.github/workflows/build-oneplus-kernel.yml`（一加专线）

这是独立于 GKI 通线的 OnePlus 构建流程（来自 OnePlus-Actions 思路）：

- 输入项是 CPU 平台 + 机型 + Android/KMI + 功能开关
- 拉的是 oneplus 对应 manifest（非 gki-kernel 那条公共线）
- 同样会注入 SukiSU + SUSFS + 可选 BBG / KPM / lz4kd
- 最后输出 AnyKernel3 包与部分 ko

> 如果你主要做一加机型，重点读这个；普通 GKI 主线看 `gki-kernel.yml`。

## 5. 一次完整编译在仓库中的“数据流”

你可以把它看成以下链路：

1. `test_release.yml` 接收用户参数
2. 将参数传给 `kernel-a**` 版本矩阵
3. 矩阵调用 `gki-kernel.yml` 实际编译并上传 artifact
4. `check-build-status` 审核结果是否可发布
5. `trigger-release` 下载全部 artifact，生成 tag，创建预发布

这就是本项目“构建系统”的核心闭环。

## 7. 你后续二改最常见的改法（实用）

### 7.1 只编你需要的内核版本

在 `kernel-a*.yml` 的 `matrix.include` 中注释不需要版本，保留你设备可用的 sublevel。

### 7.2 想减少编译时间

- 收缩 matrix
- 合理使用 ccache
- 保守打开 O2/LTO（避免不稳定）

### 7.3 想增强稳定性

- 优先 Stable 分支
- 新补丁先单版本验证（如先 a14-6.1）
- release 前先手动触发一次最小集测试

## 9. 结论（给你一句话版本）

这个项目本质是一个“**面向 GKI/SukiSU 生态的可参数化内核流水线**”：

- `test_release.yml` 负责调度
- `kernel-a*.yml` 负责版本矩阵
- `gki-kernel.yml` 负责实际构建
- `get-manager.yml + trigger-release` 负责版本信息与发布闭环

你后续维护时，**80% 工作都在 `gki-kernel.yml` 与 `kernel-a*.yml` 两类文件**。
